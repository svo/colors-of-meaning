# Feature: Real Compression Comparison (Matched Rate-Distortion)

## Overview
The repository claims color vector quantization (VQ) compresses documents, but the comparison is not honest and the three methods are not on the same axis. Three concrete defects:

1. **Fabricated original size.** `application/use_case/compress_document_use_case.py` (`_compute_compression_ratio`, ~line 71) computes `original_bits = num_tokens * 8 * 10`, an invented 80-bits-per-token constant with no basis in the source data, then divides by a bit *count* to print a "compression ratio". The number is meaningless.
2. **Non-decodable bit accounting, no shrinking artifact.** `_compute_rle_bits` (~lines 44-67) sums per-run `ceil(log2(color))`-width codes that are **not self-delimiting**, so the stream cannot be decoded; it is a bespoke bit *estimate*, never a real artifact. Worse, the only thing the `compress` CLI actually writes is a pickle of the full `List[ColoredDocument]` — including each document's `color_sequence` — so the "compressed" file does **not shrink** (`domain/model/colored_document.py` stores the full integer color sequence; `interface/cli/compress.py` `_run_vq_analysis` loads/keeps it whole).
3. **VQ reports no reconstruction error at all,** while gzip and PQ report MSE in embedding space (`infrastructure/ml/gzip_compression_baseline.py`, `infrastructure/ml/pq_compression_baseline.py`). The three methods therefore cannot be plotted on a shared rate-distortion plane. Separately, PQ's k-means is fit and predicted on the **same** `subspace_data` (`pq_compression_baseline.py` ~lines 43-51), so its distortion is in-sample and optimistically biased.

This feature puts color-VQ on the **same rate-distortion axis** as gzip and PQ: a single `CompressionBaseline` port that every method implements, a **genuine reconstruction error** for color-VQ defined as the **mean ΔE (CIE76) between each original Lab color and its dequantized codebook color**, a compressed artifact that **actually shrinks** (store fixed-width quantized codes plus the palette, not the full color sequence in a pickle), removal of the fabricated `original_bits` constant and the non-decodable variable-width RLE, and a **train/test split** for PQ's k-means so its distortion is out-of-sample. Bit accounting becomes uniform: `compressed_size_bits = num_codes * ceil(log2(num_bins))` (+ palette overhead), `original_size_bits = num_colors * 3 * <float-bits>`.

**Depends on `001-p0-1-lab-emd-distance`** (establishes the perceptual ΔE / Lab-geometry foundation; the same CIE76 ΔE in `shared/lab_utils.py:delta_e` is reused here as the reconstruction-distortion metric) and **`007-p1-1-learned-vq-codebook`** (a learned `ColorCodebook` is the palette whose dequantization error this feature measures; with the uniform grid the comparison is still valid but the distortion reflects ~88% out-of-gamut bins, which is exactly the rate-distortion penalty this feature must surface).

## User Stories
- As a researcher, I want color-VQ, gzip, and PQ reported with a real reconstruction error and a real bit count so that I can place all three on one rate-distortion plane and make an honest compression claim.
- As a researcher, I want color-VQ's distortion expressed as mean perceptual ΔE between original and dequantized Lab colors so that the error is in the units the codebook is built to preserve, not an absent or arbitrary figure.
- As a researcher, I want the "compressed" artifact to actually be smaller than the original so that the reported compression ratio corresponds to a file that genuinely shrank.
- As a maintainer, I want the fabricated `original_bits = num_tokens*8*10` constant and the non-decodable variable-width RLE removed so that no invented number reaches a report.
- As a researcher, I want PQ's k-means fit on a train split and scored on a held-out split so that its reconstruction error is out-of-sample and comparable to a deployable codec.

## Acceptance Criteria
- [ ] Given a sequence of Lab colors quantized against a `ColorCodebook`, when color-VQ reconstruction error is computed, then it equals the **mean ΔE (CIE76, `shared/lab_utils.delta_e`)** between each original Lab color and `codebook.get_color(quantized_index)`, and is `0.0` when every original color is already a codebook centroid.
- [ ] Given the same Lab colors quantized against a **coarser** codebook (fewer bins), when reconstruction error is computed, then it is **greater** than (or equal to) the error against a finer codebook — distortion rises as rate falls.
- [ ] Given a color-VQ compression run, when the compressed size is computed, then `compressed_size_bits = num_codes * ceil(log2(num_bins)) + palette_overhead_bits` and is **strictly less** than `original_size_bits` for a non-trivial input (the artifact shrinks).
- [ ] Given a color-VQ compression run, when the original size is computed, then it is `num_colors * 3 * <float-component-bits>` (the raw Lab triples), **not** `num_tokens * 8 * 10`.
- [ ] Given the codebase after this change, when `application/use_case/compress_document_use_case.py` is read, then neither `original_bits = num_tokens * 8 * 10` nor the variable-width per-run `log2(color)` RLE accounting exists anywhere.
- [ ] Given color-VQ, gzip, and PQ, when each is run through `CompressionComparisonUseCase`, then all three return a `CompressedResult` with a populated `reconstruction_error` (color-VQ in ΔE units, gzip/PQ in their existing MSE units, documented as different distortion measures), `compressed_size_bits`, `original_size_bits`, and `compression_ratio`.
- [ ] Given PQ, when `compress` is called, then k-means is `fit` on a **train split** of the rows and `predict`/scored on a **disjoint held-out split**, and the reported `reconstruction_error` is the held-out distortion (no longer in-sample on the full matrix).
- [ ] Given a degenerate input too small to split (e.g. one row), when PQ compresses it, then it falls back deterministically (documented behaviour, e.g. fit==score on the single available split) without raising.
- [ ] Given the `compress` CLI in either mode (color-VQ analysis, `--compare-baselines`), when it runs, then it prints reconstruction error and bits-per-unit for every method through the unified port and does not pickle the full `color_sequence` as the "compressed artifact".
- [ ] Given the full quality gate, when `tox` runs, then all 8 gates pass, coverage is 100%, every new test has one logical assertion (ML/numerical may group related asserts) named `test_should_..._when_...`, no comments exist, the domain layer stays pure, and `pip-audit` is green (no new runtime dependency is anticipated; if one is added it is declared in `setup.cfg`/`pyproject.toml`).

## Hexagonal Layer Impact

### Domain Layer (`src/colors_of_meaning/domain/`)
- `domain/service/compression_baseline.py` — the `CompressionBaseline` ABC (`compress(embeddings) -> CompressedResult`, `name() -> str`) is the shared port that color-VQ now also implements (today only gzip/PQ do). `CompressedResult` (`compressed_size_bits`, `original_size_bits`, `reconstruction_error: Optional[float]`, `compression_ratio` property) is reused; the dead `bits_per_dimension` property (currently `return 0.0`) is either implemented to return `compressed_size_bits / num_colors` (renamed to a meaningful `bits_per_unit`) or removed — decision in Open Questions. A small extension to the port may be needed so a color codec can accept Lab colors rather than only an embedding matrix (see Open Questions and the application note below); any extension stays a pure abstract signature with no `numpy`/`sklearn`/`ot`/`torch` body.
- `domain/model/colored_document.py` — the reconstruction-error definition needs the *original* Lab colors and the quantized indices. The document already carries `color_sequence` (quantized indices) and is built via `from_color_sequence`. Whether it should gain a **decode helper** (e.g. `reconstructed_colors(codebook) -> List[LabColor]` using `codebook.get_color`) is an Open Question; if added it is a pure method delegating to `ColorCodebook.get_color` with no I/O. The original *continuous* Lab colors (pre-quantization) are not stored on the document today — see Open Questions on where the un-quantized colors come from for ΔE.
- `domain/model/color_codebook.py` — read-only source of dequantization. `get_color(bin_index) -> LabColor` is the dequantize primitive and `quantize(color) -> int` the encode primitive; both are consumed unchanged. (Its O(num_bins) Python `quantize` loop is a performance note owned by `007-p1-1-learned-vq-codebook`, not vectorised here.)
- `domain/model/lab_color.py` — unchanged; consumed by `delta_e`.

### Application Layer (`src/colors_of_meaning/application/`)
- `application/use_case/compress_document_use_case.py` — substantially rewritten. **Remove** `_compute_compression_ratio`'s fabricated `original_bits = num_tokens * 8 * 10` and the entire `_compute_rle_bits` variable-width accounting. The use case becomes an honest color-VQ codec orchestrator: given a document's quantized `color_sequence` and the `ColorCodebook`, it computes `compressed_size_bits = len(color_sequence) * ceil(log2(num_bins)) + palette_bits`, `original_size_bits = num_original_colors * 3 * float_bits`, `compression_ratio = original/compressed`, and `reconstruction_error = mean ΔE` over original-vs-dequantized colors. It should depend on a `ColorCodebook` (constructor-injected) and on `delta_e` from `shared` (a leaf utility, layer-legal). The batch path keeps aggregating real bit totals.
- `application/use_case/compression_comparison_use_case.py` — minimal change to its loop. Today `bits_per_token = compressed_size_bits / num_sentences`; the per-row "token" denominator is fine for gzip/PQ but color-VQ's natural unit is colors/codes. Either keep the row-denominator and document it, or generalise the denominator the result carries (Open Question). The use case must report `reconstruction_error` for **all** methods, including color-VQ once color-VQ is exposed as a `CompressionBaseline` (preferred) so the comparison table has a uniform row per method.

### Infrastructure Layer (`src/colors_of_meaning/infrastructure/`)
- `infrastructure/ml/pq_compression_baseline.py` — fix the in-sample fit. Split rows into a train slice and a disjoint held-out slice (seeded, e.g. via a configured/np default `seed`), `kmeans.fit(train_subspace)`, then `kmeans.predict(holdout_subspace)` and accumulate squared error on the **held-out** rows only; `reconstruction_error` becomes held-out distortion. Handle the degenerate single-row case deterministically. `compressed_size_bits`/`original_size_bits` accounting stays float32-based and consistent (note: report the codec rate, i.e. codes for the scored rows, consistently with color-VQ and gzip).
- `infrastructure/ml/gzip_compression_baseline.py` — reference implementation, essentially unchanged; it already does a genuine `gzip.decompress` round-trip and reports real MSE, real `compressed_size_bits`, real `original_size_bits`. It is the template the color-VQ baseline mirrors structurally. (Optionally: confirm its MSE-vs-ΔE distortion unit is documented as distinct from color-VQ's ΔE in the comparison output.)
- **Color-VQ as a baseline (preferred shape):** add `infrastructure/ml/color_vq_compression_baseline.py` (or fold the logic into the use case if a port-shape mismatch makes a baseline awkward — see Open Questions) implementing `CompressionBaseline`, fit/using the injected `ColorCodebook`, computing ΔE distortion via `shared/lab_utils.delta_e` and the fixed-width code accounting. This is the file that puts color-VQ on the same axis and is the only genuinely new src file under consideration; PREFER EDITING the use case if the port can express it cleanly.

### Interface Layer (`src/colors_of_meaning/interface/`)
- `interface/cli/compress.py` — both `_run_vq_analysis` and `_run_baseline_comparison` updated. Stop treating the pickled `List[ColoredDocument]` as the "compressed artifact"; if an artifact is written it is the fixed-width codes + palette (the thing that actually shrinks). Print `reconstruction_error` (ΔE for color-VQ) alongside the existing ratio/bits columns, and route color-VQ through the unified path so the table has gzip, PQ, and color-VQ rows with comparable columns. Construct the color-VQ codec with an injected `ColorCodebook` loaded via `FileColorCodebookRepository` (the same loader the other CLIs use).
- No FastAPI controller/route/DTO changes — compression is CLI-only today; the query controller is unmounted dead code handled in `011-p2-1-mount-query-retire-coconut`.

### Shared Layer
- `shared/lab_utils.py` — `delta_e(lab1, lab2) -> float` (CIE76 Euclidean Lab, ~line 67) is **now actually called** as the color-VQ reconstruction-distortion metric (the roadmap notes it is currently defined but never invoked in `src/`). No change to its body.
- `shared/synesthetic_config.py` — if a `seed` is threaded into PQ's train/test split, it is read from the existing config seed (the project already carries `seed: 42`); add a `compression`/codec field only if bit-accounting parameters (e.g. float-component bits, palette-overhead policy) need to be configurable — otherwise no change. Decision in Open Questions.

## API Contracts
None. No new or modified FastAPI endpoints; compression has no HTTP surface in this step.

## CLI Impact
No new command. `compress` (`interface/cli/compress.py`) behaviour changes: it reports a genuine reconstruction error per method (ΔE for color-VQ, MSE for gzip/PQ, labelled as distinct distortion units), genuine shrinking-artifact bit counts, and removes the fabricated-ratio output. The `--compare-baselines` table gains a color-VQ row on the same axis as gzip and PQ. The default color-VQ mode no longer presents the full-`color_sequence` pickle as evidence of compression. No new user-facing flags are required unless a `--seed`/codec parameter is surfaced (Open Question).

## Dependency Injection
- **CLI factory wiring (constructor injection):** `compress.py` builds the color-VQ codec with a `ColorCodebook` loaded from `FileColorCodebookRepository` and injects it (and any `seed`) into the codec / `CompressDocumentUseCase`, then injects the list of `CompressionBaseline` implementations (gzip, PQ, color-VQ) into `CompressionComparisonUseCase` — exactly the constructor-injection pattern the CLI already uses for baselines.
- **Abstract → concrete mapping:** `CompressionBaseline` → `{GzipCompressionBaseline, PQCompressionBaseline, ColorVqCompressionBaseline}`; `CompressDocumentUseCase(codebook=..., seed=...)`.
- **API Lagom container:** `interface/api/main.py` registers nothing for compression (no API consumer); no change.
- **New dependency:** none anticipated — ΔE uses existing `shared/lab_utils`, PQ already uses `sklearn`, gzip is stdlib. If a helper is pulled in it is declared in `setup.cfg`/`pyproject.toml` and must pass `pip-audit`.

## Observability
- Emit one structured log entry (with `correlation-id`) per compression run summarising method, `num_codes`/`num_colors`, `compressed_size_bits`, `original_size_bits`, `compression_ratio`, and `reconstruction_error`, so a run's rate-distortion point is auditable. Do not log per-color inside the ΔE accumulation loop (hot path).
- For PQ, log once that the held-out split was used (train size, holdout size, seed) so the out-of-sample distortion is reproducible from logs.
- No new tracing spans beyond any existing use-case tracing; no per-call metrics in the inner distortion loop.

## Open Questions
- **Bit-accounting method.** Confirm the canonical formula for the shrinking artifact: is `compressed_size_bits = num_codes * ceil(log2(num_bins)) + palette_overhead`, with `palette_overhead = num_bins * 3 * float_bits` counted once (so per-corpus the palette amortises) or excluded as shared side-information? And is `original_size_bits = num_colors * 3 * 32` (float32 Lab triples) the right "original", or should it be the raw text / the 384-dim embedding (which is what gzip/PQ compress)? The three methods compress *different* originals today (color-VQ: colors; gzip/PQ: embeddings) — decide whether to normalise originals so the rate-distortion axes are truly comparable, or to report each on its own clearly-labelled axis.
- **Reconstruction needs a decode path on `ColoredDocument`?** Computing mean ΔE needs the *original continuous* Lab colors and their dequantized codebook colors. `ColoredDocument.color_sequence` stores only quantized indices; the pre-quantization continuous colors are not currently retained. Decide whether (a) `ColoredDocument` gains a pure `reconstructed_colors(codebook)` decode helper and the original continuous colors are threaded in from the encode step, (b) the codec recomputes ΔE at encode time before discarding the continuous colors (so the use case needs the continuous colors as input, not the document), or (c) distortion is defined index-to-centroid only where the un-quantized color is unavailable. This choice determines whether a domain-model change is in scope or deferred.
- **PQ split policy / determinism.** Fixed fraction (e.g. 80/20) vs configured? Seeded via the existing `synesthetic_config` `seed` or an explicit `--seed`? Behaviour for inputs too small to split must be pinned (fall back to in-sample on the single split, documented) so coverage and determinism hold.
- **`bits_per_dimension` stub.** `CompressedResult.bits_per_dimension` currently returns a hardcoded `0.0`. Implement it as a real bits-per-unit (and rename for honesty) or delete it — confirm which, since 100% coverage forbids leaving a decorative dead branch.
