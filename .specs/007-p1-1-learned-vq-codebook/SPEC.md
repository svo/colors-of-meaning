# Feature: Learned VQ Codebook

## Overview
The color codebook is currently built by `ColorCodebook.create_uniform_grid(bins_per_dimension)` (`domain/model/color_codebook.py` lines 43-57), which lays a uniform 16x16x16 lattice across the full nominal Lab box (L in [0,100], a/b in [-128,127]). Because the projector only ever emits colors inside a small, data-shaped sub-region of that box, the overwhelming majority of the 4,096 cells fall on out-of-gamut or never-visited coordinates, so quantization wastes most of its resolution and the term "vector quantization" is not honest. This feature adds a learned palette: a new `infrastructure/ml/learned_color_codebook_factory.py` fits `MiniBatchKMeans(n_clusters=num_bins)` on the **projected** Lab colors produced by a trained `ColorMapper`, and constructs a `ColorCodebook` whose `colors` are the (clamped) `cluster_centers_`. The factory lives in infrastructure to keep the domain free of sklearn; `ColorCodebook` already accepts any `List[LabColor]` of length `num_bins`, so no domain entity change is required. The same change vectorises `ColorCodebook.quantize` (lines 19-29) from its O(num_bins) Python loop into a single batched nearest-centroid lookup (`argmin` over `scipy.spatial.distance.cdist`, or `kmeans.predict`). `create_uniform_grid` is retained as an A/B reconstruction baseline (this is the same uniform-vs-learned comparison surfaced by `009-p1-3-real-compression-comparison`). This work depends on `002-p0-2-structure-preserving-training` to make the projected Lab distribution meaningful (clustering noise yields a useless palette) and benefits `001-p0-1-lab-emd-distance`, whose Wasserstein retrieval over color histograms is more discriminative when bins land on real density.

## User Stories
- As a researcher validating the Colors of Meaning thesis, I want the codebook fitted to the actual projected Lab distribution so that quantization spends its 4,096 bins on colors the projector really emits, making "vector quantization" honest rather than a uniform lattice that wastes ~88% of bins.
- As a researcher, I want to choose at train time between the uniform baseline and the learned codebook so that I can run an A/B reconstruction comparison (uniform grid versus k-means palette) on identical data.
- As a pipeline owner, I want `ColorCodebook.quantize` vectorised to a batched nearest-centroid lookup so that encoding documents into color histograms is not bottlenecked by an O(4,096) Python loop per color.
- As a maintainer guarding hexagonal boundaries, I want the k-means fit to live in infrastructure so that the domain entity stays pure (no sklearn/torch import) while still producing a standard `ColorCodebook`.

## Acceptance Criteria
- [ ] Given a trained `ColorMapper` and a set of embeddings, when `LearnedColorCodebookFactory.build(...)` runs, then it returns a `ColorCodebook` whose `num_bins == n_clusters` and whose `colors` are the k-means `cluster_centers_` mapped to valid in-range `LabColor` instances.
- [ ] Given the factory fits `MiniBatchKMeans`, when it is constructed, then `n_clusters == num_bins`, `random_state == seed`, and the fit is performed on the projected Lab array obtained via the `ColorMapper` (not on raw 384-dim embeddings).
- [ ] Given the same embeddings, the same trained mapper, and the same seed, when `build(...)` is called twice, then the two resulting codebooks have identical palettes (deterministic k-means under the fixed `random_state`).
- [ ] Given any centroid coordinate returned by k-means that falls outside the Lab box, when the factory constructs `LabColor` instances, then the coordinate is clamped (via `LabColor.clamp` / `shared.lab_utils` range clamping) so `LabColor.__post_init__` does not raise.
- [ ] Given a `ColorCodebook` (uniform or learned), when `quantize(color)` is called, then it returns the index of the nearest palette color by Euclidean Lab distance, identical to the previous loop result, via a vectorised nearest-centroid computation rather than a per-color Python loop.
- [ ] Given fewer unique projected colors than `num_bins`, when the factory builds the codebook, then the degenerate case is handled deterministically (defined behaviour: reduce `n_clusters` to the number of distinct points, or pad, so the resulting `ColorCodebook` still satisfies `len(colors) == num_bins`).
- [ ] Given the train CLI, when the codebook-mode flag selects the learned codebook, then `TrainColorMappingUseCase.execute(...)` builds the palette from data via the injected factory; when it selects uniform, then `ColorCodebook.create_uniform_grid(bins_per_dimension)` is used exactly as today (default behaviour unchanged unless the flag is set).
- [ ] Given the domain layer, when a `pytest-archon` boundary test runs, then `domain/` imports no sklearn (and no torch); the k-means dependency is confined to `infrastructure/ml/`.
- [ ] Given the full suite, when `tox` runs, then all gates pass (flake8, black, bandit, semgrep, xenon, radon, mypy, pip-audit), coverage stays 100%, every new test has one logical assertion and is named `test_should_..._when_...`, and no comments are introduced.

## Hexagonal Layer Impact

### Domain Layer (`src/colors_of_meaning/domain/`)
- `domain/model/color_codebook.py`: `quantize` (lines 19-29) is vectorised from the O(num_bins) Python loop into a single batched nearest-centroid lookup over the palette (precompute a `(num_bins, 3)` numpy array of Lab coordinates once, then `int(argmin)` over Euclidean distances to the query color); `_euclidean_distance` (lines 36-41) may be removed or retained as the scalar reference depending on the vectorised form. Behaviour is bit-for-bit equivalent on ties to the existing first-minimum semantics (`<`, not `<=`), so existing quantize tests must still pass. `create_uniform_grid` (lines 43-57) is kept verbatim as the A/B baseline. The entity stays a frozen dataclass over `List[LabColor]` and imports no sklearn/torch (numpy is already used here, line 3, and is allowed in the domain). The constructor invariant (`len(colors) == num_bins`, `num_bins > 0`) is unchanged and is what the factory must satisfy.
- `domain/model/lab_color.py`: unchanged; its `clamp()` (lines 26-30) and range invariants are reused by the factory to keep out-of-gamut centroids valid.
- `domain/repository/color_codebook_repository.py`: unchanged; the learned codebook is persisted through the existing `save/load/exists/delete` port.
- `domain/service/color_mapper.py`: unchanged; the factory consumes the existing `embed_batch_to_lab(embeddings) -> List[LabColor]` port (line 15) as its source of projected colors. No new domain port is introduced (the factory is an infrastructure concern, not a domain service).

### Application Layer (`src/colors_of_meaning/application/`)
- `application/use_case/train_color_mapping_use_case.py`: the hardcoded `ColorCodebook.create_uniform_grid(...)` at line 32 becomes the default branch of a codebook-mode decision. The use case gains an optional injected collaborator (a learned-codebook factory abstraction) and a parameter on `execute(...)` selecting uniform versus learned; when learned is selected it delegates palette construction to the factory using the already-trained `self.color_mapper` plus the `embeddings` it already receives, then `self.codebook_repository.save(codebook, codebook_name)` exactly as today. The use case still imports only domain symbols: the factory is referenced through a domain-defined abstraction (an `ABC` in `domain/`, or the existing-pattern equivalent) so the application never imports `infrastructure`. No business logic (the k-means math) moves into application.

### Infrastructure Layer (`src/colors_of_meaning/infrastructure/`)
Primary scope.
- New `infrastructure/ml/learned_color_codebook_factory.py`: implements the codebook-factory abstraction. `build(embeddings, num_bins, seed)` obtains projected colors via `self.color_mapper.embed_batch_to_lab(embeddings)`, stacks them into a `(n_samples, 3)` numpy array, fits `MiniBatchKMeans(n_clusters=num_bins, random_state=seed, ...)`, then maps each row of `kmeans.cluster_centers_` to a clamped `LabColor` and returns `ColorCodebook(colors=..., num_bins=num_bins)`. Follows the established sklearn usage already present in the codebase: `pq_compression_baseline.py` (lines 44-50) uses `MiniBatchKMeans(n_clusters=..., random_state=42, n_init=1, batch_size=min(256, num_samples))` with `.fit()`, `.predict()`, and `.cluster_centers_[codes]`; `structured_pytorch_color_mapper.py` line 211 uses `KMeans(n_clusters=..., random_state=42, n_init=10)`. The factory injects the trained `ColorMapper` (constructor injection) so it never re-trains and never touches files. Degenerate fewer-points-than-bins handling lives here.
- `infrastructure/persistence/file_color_codebook_repository.py`: unchanged; it pickles any `ColorCodebook` (lines 16-19), so the learned palette persists without modification.

### Interface Layer (`src/colors_of_meaning/interface/`)
- `interface/cli/train.py`: `TrainArgs` (lines 36-42) gains a codebook-mode flag (e.g. `codebook_mode: str = "uniform"` with values `uniform`/`learned`, or a `learned_codebook: bool = False`), defaulting to today's uniform behaviour. `_execute_training` (lines 145-166) constructs the `LearnedColorCodebookFactory` (wrapping the already-built `color_mapper`) via a factory function and injects it into `TrainColorMappingUseCase`, and passes the chosen mode through to `use_case.execute(...)`. The print line "Codebook saved to artifacts/codebooks/{output_codebook}.pkl" (line 165) is retained; an additional line may state which mode produced it. No API surface is touched.

### Shared Layer
- `shared/lab_utils.py`: reused, not necessarily modified. The factory clamps centroid coordinates using `LabColor.clamp()` or `scale_to_lab_range`/explicit `np.clip` against the documented Lab bounds; `delta_e` (lines 67-71, numpy CIE76) remains the reference Euclidean Lab distance consistent with `ColorCodebook._euclidean_distance`. If a single batched clamp helper is wanted it would be added here (numpy-only, no new dependency); whether to add it is an Open Question.
- `shared/synesthetic_config.py`: `CodebookConfig` (lines 17-19: `bins_per_dimension=16`, `num_bins=4096`) and `TrainingConfig.seed` are consumed unchanged; the factory's `num_bins` comes from `CodebookConfig.num_bins` and its `seed` from `TrainingConfig.seed`. Adding a persisted `codebook.mode` field to `CodebookConfig` (with `from_yaml`/`to_yaml` round-trip coverage at lines 86, 97) versus keeping mode CLI-only is an Open Question.

## API Contracts
No changes. This feature introduces no new endpoints and modifies no Pydantic DTOs. The learned codebook is reachable only through the `train` CLI command; the FastAPI surface (`interface/api/main.py`, controllers) is untouched. Endpoints continue to return Pydantic DTOs as mandated.

## CLI Impact
`train.py` gains one codebook-mode flag (proposed `--codebook-mode {uniform,learned}`, default `uniform`) so existing invocations are byte-for-byte unchanged. When `learned` is selected, the codebook is fitted to the projected Lab colors of the training embeddings (the same embeddings already encoded at line 135) using the trained mapper; the saved artifact name/path is unchanged. No other CLI command changes here; `eval`/`encode`/`compare` simply load whatever codebook was saved through the existing repository. The uniform-versus-learned reconstruction A/B reporting is owned by `009-p1-3-real-compression-comparison`; this step only makes both palettes producible from the same data.

## Dependency Injection
Consistent with project reality. The CLI wires via factory functions plus constructor injection (e.g. `TrainColorMappingUseCase(color_mapper=..., codebook_repository=...)` in `interface/cli/train.py`); this feature adds the `LearnedColorCodebookFactory` as a third constructor-injected collaborator built by a factory function in `train.py` and passed into the use case, exactly mirroring how `FileColorCodebookRepository` is constructed today (line 152). The application references the factory through a domain-defined abstraction (`ABC` in `domain/`) so it never imports infrastructure. The API runtime's Lagom `Container()` in `interface/api/main.py` is untouched because training is not exposed via API; if the factory ever needs API-side resolution it would be registered there, but that is out of scope. No direct instantiation inside the use case.

## Observability
Preserve the existing `print` cadence in `train.py` (lines 164-165). Add a structured log line (with `correlation-id`, JSON, consistent with project observability rules) recording the codebook mode, `num_bins`, number of projected colors fed to k-means, and the achieved inertia / mean reconstruction error of the learned palette so the ~88%-bins-reclaimed claim is measurable rather than asserted. Emit a metric for codebook fit duration and reconstruction error (gauge/histogram). No secrets are logged. The vectorised `quantize` must not regress any existing logging.

## Open Questions
- Where do the projected colors come from at codebook-build time? Proposed: project the same training `embeddings` (already in hand at `train.py` line 135) through the just-trained `self.color_mapper.embed_batch_to_lab(...)`. Alternatives: project a held-out split, or persist the projected Lab array during training to avoid a second forward pass. Should the factory call the mapper itself (mapper injected) or receive a precomputed `(n,3)` array?
- Train CLI flag to choose uniform versus learned: exact name and type (`--codebook-mode {uniform,learned}` versus boolean `--learned-codebook`), default (must stay `uniform` to preserve current behaviour), and whether the chosen mode is also persisted into `CodebookConfig`/the YAML (requiring `from_yaml`/`to_yaml` round-trip tests) or kept CLI-only.
- `MiniBatchKMeans` `n_init`: the ROADMAP text says `n_init='auto'`, but the existing in-repo precedent (`pq_compression_baseline.py` line 45) uses `n_init=1` and `structured_pytorch_color_mapper.py` line 211 uses `n_init=10`. Which value, and is `MiniBatchKMeans` (memory-friendly at 4,096 clusters) preferred over full `KMeans` for determinism? Confirm `batch_size` (PQ uses `min(256, num_samples)`).
- Vectorised `quantize` implementation: `scipy.spatial.distance.cdist` + `argmin` (scipy already a dependency) versus caching `kmeans.predict` on the factory (couples the domain entity to a fitted estimator â€” likely rejected to keep the domain pure). Confirm tie-breaking matches the current first-minimum (`<`) semantics so existing tests pass unchanged.
- Degenerate cases: behaviour when unique projected colors `< num_bins` (shrink `n_clusters` and break the `len(colors) == num_bins` invariant, versus pad to `num_bins`), and when `embeddings` is empty. Define and test both.
- Centroid clamping: rely on `LabColor.clamp()` per row, or add a batched numpy clamp helper to `shared/lab_utils.py`? Decide whether clamping should be logged when it changes a centroid (signals out-of-gamut k-means output).
- Minimum training-set size for a meaningful 4,096-color palette, and the dependency on `002-p0-2-structure-preserving-training`: until the projector emits a structured Lab distribution, the learned palette may be no better than uniform â€” how should the A/B baseline (`009`) gate acceptance of "honest VQ"?
