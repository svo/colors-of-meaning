# Feature: Honest Interpretable Mapper

## Overview
The structured projector promises that its three perceptual axes carry real meaning — hue encodes semantic clusters, lightness encodes sentiment, chroma encodes concreteness — but today none of them do. In `infrastructure/ml/structured_pytorch_color_mapper.py`, `_derive_lightness_targets` (lines 221-232) sets lightness to the normalised **mean** of the embedding dimensions, `_derive_chroma_targets` (lines 234-245) sets chroma to the normalised **variance**, and `_derive_hue_targets` (lines 205-219) assigns each KMeans cluster an arbitrary angle `2π·label/K` (lines 214-217) so adjacent hues are unrelated to adjacent meanings. The mean and variance of a sentence-transformers embedding are not sentiment and not concreteness; the cluster-index angle is not a semantic ordering. This feature replaces all three proxies with honest signals: lightness driven by **real sentiment** (IMDB labels when available, otherwise a sentiment signal derived from the texts), chroma driven by a **real concreteness signal** (Brysbaert-style concreteness norms looked up per text via a new infrastructure lexicon adapter), and hue clusters **ordered along a 1-D embedding of the cluster centroids** so hue adjacency approximates semantic adjacency.

The architectural obstacle is that the `ColorMapper` port exposes only `train(embeddings, epochs, learning_rate)` — embeddings carry no sentiment, no text, and no concreteness. The supervised mapper already solved the identical problem by adding a `set_training_labels(labels)` setter consumed inside `train`; this feature follows that exact precedent, adding setters on `StructuredPyTorchColorMapper` to inject the per-sample sentiment scores and the per-sample texts (for concreteness lookup) before training, with deterministic fallbacks when they are absent so existing call sites keep working. The port itself does not change, so the domain and application layers stay untouched.

This step substantiates the blog's interpretability claim and is the precondition for `010-p1-4-ablations`, which measures the interpretability-versus-performance tradeoff the article explicitly poses (honest axes will plausibly cost accuracy versus the unconstrained `PyTorchColorMapper`, and the ablation can only quantify that once the axes are honest). It depends on P0: it relies on the structure-preserving objective and structured loss of `002-p0-2-structure-preserving-training`, the stratified non-degenerate sampling of `003-p0-3-shuffle-stratify-sampling` (the IMDB head-slice in `imdb_dataset_adapter.py` lines 17-19 yields single-class splits that would make sentiment-driven lightness uniform), the determinism and structure metric of `005-p0-5-determinism-structure-metric`, and the end-to-end harness of `006-p0-6-end-to-end-agnews-table`.

## User Stories
- As a researcher defending the interpretability claim, I want lightness to track real sentiment so that dark-to-light actually reads negative-to-positive instead of reflecting the arithmetic mean of embedding dimensions.
- As a researcher, I want chroma to track a real concreteness signal so that vivid-versus-muted colors correspond to concrete-versus-abstract text instead of embedding variance.
- As a researcher, I want hue clusters ordered so neighbouring hues are semantically neighbouring, so that the color wheel reads as a semantic continuum rather than an arbitrary `2π·idx/K` shuffle.
- As the owner of `010-p1-4-ablations`, I want the structured mapper to expose honest, separately attributable axes so that I can measure how much accuracy the interpretability constraint costs against the unconstrained baseline.
- As a maintainer, I want the existing `ColorMapper.train(embeddings, epochs, learning_rate)` port and all current call sites to keep working without supplying sentiment or text, so that honest signals are opt-in and the change is backward compatible.
- As a pipeline owner, I want the concreteness lexicon to be a bundled, offline package resource so that training stays deterministic and unit tests make no network calls.

## Acceptance Criteria
- [ ] Given per-sample sentiment scores set on the structured mapper, when lightness targets are derived, then lightness is a monotonic increasing function of sentiment (more positive sentiment yields higher L), not the normalised mean of embedding dimensions.
- [ ] Given two batches identical except that one has uniformly more positive sentiment, when lightness targets are derived for each, then the more-positive batch has higher mean lightness.
- [ ] Given IMDB binary labels (0 negative, 1 positive) supplied as the sentiment source, when lightness targets are derived, then negative-labelled samples map to a low-L band and positive-labelled samples to a high-L band, with all values inside `[0, 100]`.
- [ ] Given per-sample texts set on the structured mapper, when chroma targets are derived, then chroma is a monotonic increasing function of the concreteness signal looked up from the lexicon, not the normalised variance of the embedding.
- [ ] Given a concrete text and an abstract text scored against the bundled concreteness norms, when chroma targets are derived, then the concrete text receives higher chroma than the abstract text, with all values inside `[0, max_chroma]`.
- [ ] Given a text whose tokens are entirely absent from the concreteness lexicon, when its concreteness is scored, then a defined neutral fallback concreteness is returned (no error, no NaN) and the resulting chroma is the configured midpoint.
- [ ] Given the cluster centroids of a trained KMeans, when hue angles are assigned, then clusters are ordered by a deterministic 1-D ordering of their centroids (e.g. projection onto the principal axis) and mapped to angles in that order, so adjacent angles correspond to adjacent centroids rather than to adjacent cluster indices.
- [ ] Given two embeddings whose centroids are nearest neighbours in embedding space, when hue angles are assigned, then their hue angles are adjacent on the wheel (closer than a far-apart pair), which would fail under the prior `2π·label/K` index ordering.
- [ ] Given no sentiment scores and no texts are set, when the structured mapper trains, then it falls back to defined deterministic targets (documented neutral defaults) and existing call sites that pass only embeddings continue to train without error.
- [ ] Given the bundled concreteness norms resource, when any unit test runs, then concreteness is resolved entirely offline through `resources.get_resource_path` with no network call and no dataset download.
- [ ] Given the chosen sentiment source mode, when configured via `StructuredMapperConfig`, then `from_yaml`/`to_yaml` round-trips the new fields unchanged.
- [ ] Given the full suite, when `tox` runs, then all gates pass, coverage stays 100%, every new test has one logical assertion and is named `test_should_..._when_...`, and no comments are introduced.

## Hexagonal Layer Impact

### Domain Layer (`src/colors_of_meaning/domain/`)
No changes. The `ColorMapper` port in `domain/service/color_mapper.py` keeps its signatures (`embed_to_lab`, `embed_batch_to_lab`, `train(embeddings, epochs, learning_rate)`, `save_weights`, `load_weights`); honest target derivation and the optional sentiment/text setters are infrastructure implementation details, mirroring how `set_training_labels` on the supervised mapper sits outside the port. `QuantizedColorMapper` is unaffected. `LabColor` (`domain/model/lab_color.py`) is unchanged and remains the source of truth for valid ranges (L in `[0, 100]`, a/b in `[-128, 127]`, with `clamp()` applied); derived lightness must stay within `[0, 100]` and derived chroma within `[0, max_chroma]` so the `__post_init__` validation never raises. Domain stays pure (no torch, no sklearn, no I/O). A concreteness norm is a numeric lexical attribute, not a business entity, so no new domain model is introduced (revisited in Open Questions).

### Application Layer (`src/colors_of_meaning/application/`)
No changes. `TrainColorMappingUseCase.execute(...)` delegates to `color_mapper.train(embeddings=..., epochs=..., learning_rate=...)`; because honest targets are derived behind the port and the new setters are called by the CLI wiring (as `set_training_labels` already is), no orchestration change is required. The application layer must not import the concreteness adapter or torch.

### Infrastructure Layer (`src/colors_of_meaning/infrastructure/`)
Primary scope.
- `infrastructure/ml/structured_pytorch_color_mapper.py`:
  - `_derive_lightness_targets` (lines 221-232): stop using `np.mean(embeddings, axis=1)`; when per-sample sentiment scores are set, map sentiment (e.g. `[0, 1]` for IMDB labels, or a continuous polarity) monotonically onto `[0, 100]`; when absent, return the documented neutral fallback (e.g. constant 50.0, matching the existing degenerate branch).
  - `_derive_chroma_targets` (lines 234-245): stop using `np.var(embeddings, axis=1)`; when per-sample texts are set, look up each text's concreteness via the new lexicon adapter and map it monotonically onto `[0, max_chroma]`; when absent, return the documented neutral fallback (`max_chroma / 2`, matching the existing degenerate branch).
  - `_derive_hue_targets` (lines 205-219): keep KMeans clustering of L2-normalised embeddings, but replace the `2π·label/K` assignment (lines 214-217) with a deterministic 1-D ordering of `kmeans.cluster_centers_` (e.g. projection onto the first principal component of the centroids, or a TSP-free greedy nearest-neighbour chain), then assign each cluster its angle by rank in that ordering so adjacent ranks get adjacent angles. KMeans `random_state=42` is retained for determinism (reinforced by P0-5).
  - Add `set_sentiment_scores(scores)` and `set_training_texts(texts)` setters storing optional per-sample arrays/lists (analogous to `set_training_labels` on the supervised mapper); `train` and `_derive_targets` consume them when present. Constructor-injected dependency on the concreteness lexicon port (default-constructed if not provided) keeps DI honest.
- New `infrastructure/ml/brysbaert_concreteness_lexicon.py` (or `infrastructure/embedding/`/`infrastructure/dataset/` — placement is an Open Question): a concrete implementation of a new concreteness-lookup port that loads the bundled norms once and returns a per-text concreteness score (token-mean of in-lexicon words, neutral fallback for out-of-vocabulary text). Loads via `resources.get_resource_path(...)`; performs no network call.
- New bundled data resource under `src/colors_of_meaning/resources/` (e.g. `concreteness_norms.tsv` or a trimmed subset): the Brysbaert concreteness ratings (word, concreteness) used offline. `include_package_data = True` and the `resources` package mapping in `setup.cfg` already ship `resources/`; the file is added there. A tiny fixture lexicon is used in unit tests to avoid shipping/parsing the full list in the fast suite.
- A concreteness-lookup port (ABC) is defined in `domain/service/` if the lexicon is treated as a domain service abstraction (preferred for testability and DI symmetry with `ColorMapper`); see Dependency Injection and Open Questions. The interface is pure; only its implementation lives in infrastructure.

### Interface Layer (`src/colors_of_meaning/interface/`)
- `interface/cli/train.py`: `_create_color_mapper` (lines 57-95) constructs `StructuredPyTorchColorMapper` from `StructuredMapperConfig`; it gains construction of the concreteness lexicon adapter and injection into the structured mapper. `main`/`_execute_training` (lines 122-165) gains, on the structured path, calls to `set_sentiment_scores(...)` and `set_training_texts(...)` before training — exactly mirroring the existing `_configure_supervised_mapper` (lines 114-119) that calls `set_training_labels` on the supervised path. When the dataset provides sentiment labels (IMDB via `_load_supervised_data`), those become the sentiment source; otherwise the configured fallback applies. No new top-level CLI command; an optional flag may select the sentiment source mode (Open Question). No API change.

### Shared Layer
- `shared/synesthetic_config.py`: `StructuredMapperConfig` (lines 46-51) gains fields for the honesty knobs — e.g. `sentiment_source` (`"labels"` vs `"model"` vs `"none"`), `concreteness_resource` (resource filename), and any lightness/chroma band parameters — with `from_yaml` (lines 76-92) and `to_yaml` (lines 94-107) round-tripping them. Defaults preserve current behaviour where a sensible neutral fallback applies. `shared/lab_utils.py` is unaffected. `shared` stays torch-free; the concreteness file path is resolved via the existing `resources` package helper, not added to `shared`.

## API Contracts
No changes. This feature introduces no endpoints and modifies no DTOs. The FastAPI surface (`interface/api/main.py` with its Lagom `Container()`, the health/coconut/query controllers) is untouched; honest target derivation is reachable only through the `train` CLI path. The interface-layer rule that endpoints return Pydantic DTOs is not engaged because no endpoint is added or changed.

## CLI Impact
`train.py` continues to support `--mapper-type {unconstrained,structured,supervised}`. Observable behaviour change on the structured path: lightness now reflects sentiment and chroma reflects concreteness, so trained colors of positive/concrete text differ visibly from negative/abstract text; the hue wheel becomes a semantic continuum. When training the structured mapper on IMDB, sentiment is taken from the labels loaded by `_load_supervised_data`; on free-text input (`--dataset-path`) the configured fallback or sentiment-model mode applies. An optional `--sentiment-source` style flag mapped to `StructuredMapperConfig.sentiment_source` may be added (Open Question). The unconstrained and supervised paths are unchanged. The `eval` `--mapper-type` reach fix remains owned by `004-p0-4-eval-mapper-coverage-ef-fix`.

## Dependency Injection
Consistent with project reality: the CLI wires mappers via explicit factory functions (`_create_color_mapper` in `interface/cli/train.py`) and constructor-injects them into `TrainColorMappingUseCase`; the API builds a Lagom `Container()` in `interface/api/main.py` but does not touch training. The new concreteness lexicon is exposed as a port (ABC, preferably in `domain/service/`) and its concrete implementation is constructed in `_create_color_mapper` and passed into `StructuredPyTorchColorMapper`'s constructor (default-constructed if omitted, so existing direct instantiations in tests still work). The sentiment scores and texts are injected through setters at wiring time, exactly as `set_training_labels` is today. No Lagom container changes are required because training is CLI-driven; if a future API path needs the lexicon, the existing `Container()` can register the port-to-implementation binding without touching consumers.

## Observability
Preserve the existing per-epoch `print` cadence (every 10 epochs, lines 111-112). Emit, at structured-mapper setup, a one-line log of the active sentiment source and concreteness resource (no secrets, no raw text) so runs are auditable. Structured logging with `correlation-id`, metrics, and the structure-preservation / interpretability metrics are owned by `005-p0-5-determinism-structure-metric` and consumed by `010-p1-4-ablations`; this step must remain compatible with them. No comments are added; intent is carried by names (`set_sentiment_scores`, `_order_clusters_by_centroid`, `_score_concreteness`). No secrets are logged.

## Open Questions
- Concreteness-norms sourcing and licensing: the Brysbaert et al. (2014) concreteness ratings are widely used; confirm the redistribution licence permits bundling under `src/colors_of_meaning/resources/`, and decide whether to ship the full ~40k-word list, a trimmed high-frequency subset, or a download-on-first-use cache (the last would break the no-network-in-tests rule for integration unless guarded). Until confirmed, ship a small placeholder lexicon plus a tiny test fixture and document the swap.
- Whether to add a concreteness lexicon adapter under infrastructure as proposed (with a `domain/service` port) versus a lighter `shared/` utility: a port keeps DI symmetric with `ColorMapper` and isolates the file/parsing I/O in infrastructure (recommended), but adds a domain abstraction for what is arguably a lookup table; confirm the boundary call.
- Sentiment-model versus label-only: IMDB supplies gold binary labels (cleanest, fully offline, deterministic), but AG News / free text have none, so a sentiment source is needed there. Options: label-only (lightness honest only on labelled sentiment datasets, neutral fallback elsewhere), a lightweight lexicon-based polarity score (offline, no new heavy dep), or a sentence-transformers/zero-shot sentiment model (heavier, possible model download conflicting with offline tests). Recommended default: labels when present, else a lightweight offline polarity signal; pick one for the first cut.
- Any new runtime dependency (e.g. a sentiment lexicon package) must be declared in `setup.cfg` `install_requires` and pass `pip-audit`; note `pyproject.toml` currently declares no runtime dependencies (only build/tooling config), so the dependency source of truth is `setup.cfg`. Prefer a pure-data resource over a new package where possible to avoid a new `pip-audit` surface.
- Sentiment-to-lightness and concreteness-to-chroma mapping shape: linear, banded (two bands for binary labels), or sigmoid; and where the band edges / scale parameters live (`StructuredMapperConfig` versus adapter defaults). Adding config fields requires `from_yaml`/`to_yaml` round-trip coverage.
- 1-D centroid ordering method: principal-component projection of `cluster_centers_` versus a greedy nearest-neighbour chain versus angular MDS; choose the one that is deterministic under fixed seed and keeps `xenon`/`radon` complexity within limits.
- The existing `TestStructuredTargetDerivation` tests assert lightness == mean and chroma == variance behaviour shapes; these will be rewritten to assert the honest behaviour (Task List covers this) so coverage stays meaningful and 100%.
- Degenerate-batch behaviour for the new derivations (single sample, all-identical sentiment, empty-after-tokenisation text) must each have a defined neutral output and a test, consistent with the existing constant-embedding fallbacks.
